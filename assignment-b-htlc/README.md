# Assignment B - Hashed Time-Lock Contract (HTLC)
## 1. Overview

This assignment demonstrates the implementation of a Hashed Time-Lock Contract (HTLC) using Bitcoin Script in a controlled regtest environment.

An HTLC is a smart contract mechanism that allows two parties (in this case, Alice and Bob) to exchange funds in a trustless way. It ensures fairness by enforcing two important conditions:

    - A hashlock, which allows Alice to claim funds only if she provides a correct secret.

    - A timelock, which allows Bob to recover/ refund the funds if Alice does not act within a specified time.

This combination ensures that either the correct secret is revealed and funds are claimed, or the sender safely gets their funds back after a delay.

## 3. Setup and Environment

I  performed all the tests using Bitcoin Core in regtest mode.

Regtest is a local blockchain environment that allows full control over:

    - Mining blocks instantly

    - Creating wallets and transactions

    - Testing scripts without using real Bitcoin

This makes it ideal for safely experimenting with Bitcoin Script logic such as HTLCs.

### `Fee Configuration Note`

While working on the assignment, I encountered an error when trying to send a transaction:

"Fee estimation failed. Fallbackfee is disabled."

This happened because, in regtest mode, there is no real transaction activity. Bitcoin Core cannot estimate transaction fees without mempool data.To fix this, I restarted the node with a fallback fee enabled:

```bash

bitcoind -regtest -fallbackfee=0.0001 -daemon

```
This allows transactions to be created even when fee estimation is not possible.

## 5. HTLC Script Implementation

The main HTLC script is stored in: scripts/htlc_script.txt

This script defines two possible spending paths using conditional logic.

### `Script Logic`

If the correct condition is met, one of two branches is executed:

    - Alice’s path (claim funds using secret)

    - Bob’s path (refund after time delay)

### `Script Breakdown`

The script begins with:

OP_IF

This checks which branch will be executed.

#### **Alice’s Branch (Hashlock)**

OP_HASH160 OP_EQUALVERIFY
This verifies that the provided secret matches the expected hash.

<alice_pubkey> OP_CHECKSIG
This ensures that Alice signs the transaction.

For Alice to claim the funds, she must:

    - Provide the correct secret

    - Produce a valid signature

#### **Bob’s Branch (Timelock)**

OP_ELSE

OP_CHECKLOCKTIMEVERIFY OP_DROP
This enforces that the transaction can only be executed after a specific block height.

<bob_pubkey> OP_CHECKSIG
This ensures that Bob signs the transaction.

For Bob to refund the funds, he must:

    - Wait until the locktime is reached

    - Provide a valid signature

Script Ends

OP_ENDIF

This closes the conditional logic.

The locktime ensures that Bob cannot immediately refund the funds when Alice sends them.

After mining additional blocks:

This demonstrates that:
- Before the locktime, refund is not valid
- After the locktime, refund becomes possible

### `Time-Lock Interpretation`

The assignment required me to implement a 21-minute timeout, and in Bitcoin, time is typically represented using block height rather than exact minutes.
Since one block takes approximately 10 minutes,i decided to use a 2-block delay (~20 minutes)to simulate the required 21-minute timeout in the regtest environment.


## 8. Claim and Refund Logic

### **Claim (Alice Path)**

Stored in: scripts/claim_script.txt

Alice claims funds by:

    - Providing the correct secret

    - Triggering the correct branch (using OP_TRUE / 1)

    - Providing a valid signature

This satisfies the hashlock condition.

### **Refund (Bob Path)**

I have stored this in: scripts/refund_script.txt

Bob refunds funds by:

    - Waiting until the locktime is reached

    - Triggering the alternative branch (using OP_FALSE / 0)

    - Providing a valid signature

This satisfies the timelock condition.
